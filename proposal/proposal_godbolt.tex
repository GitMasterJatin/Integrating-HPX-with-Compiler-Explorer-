\section*{Abstract}

Compiler Explorer (\url{https://godbolt.org/}) is a widely popular web-based
application which provides easy access to multiple C++ compilers and
environments, allowing users to write, test, and share anything from small
C++ scripts to large CMake-based projects quickly.  Given its versatility, we
believe it would be convenient for HPX to have an integration with Compiler
Explorer.  A preliminary idea is that we will maintain our own fork of
Compiler Explorer (which is open source) and experiment with the HPX
integration locally before making the integration public through a Pull
Request to Compiler Explorer.  The result could even be constrained to an
environment somewhat similar to Compiler Explorer just for HPX, where
prospective users would experiment with quick HPX scripts before downloading,
building, and running the whole library.

I have already \textbf{built a working proof of concept}
(HPX~2.0.0 compiling and executing inside a local CE instance) and
\textbf{published the fix} at
\href{https://github.com/GitMasterJatin/Integrating-HPX-with-Compiler-Explorer-}%
{github.com/GitMasterJatin/Integrating-HPX-with-Compiler-Explorer-}.
This proposal converts that validated prototype into three
upstream Pull Requests across the HPX and CE repositories, giving 3.5~million
users the ability to try HPX parallelism in seconds, replacing a 30-minute
local build with a single checkbox.

\subsection*{Interest in HPX}

HPX implements the complete C++ standard parallel algorithm suite
(\code{hpx::for\_each}, \code{hpx::transform\_reduce}, \code{hpx::sort})
and extends it with futures, continuations, and a work-stealing runtime.
Code written with \code{hpx::execution::par} is structurally identical to
\code{std::execution::par}, making HPX the ideal teaching and prototyping tool
for modern C++ parallelism.

% ── Root Cause Analysis ──────────────────────────────────────────────────────
\subsection*{Root Cause Analysis: Why HPX Is Broken on CE}

An HPX entry already exists in CE's production configuration.  After auditing
\cefile{c++.amazon.properties} (lines~5759--5767) and tracing through CE's
build pipeline (\cefile{library\_builder.py}), I identified \textbf{six root
causes}.  The current broken entry:

% [GROUNDED: compiler-explorer/etc/config/c++.amazon.properties:5759-5767]
\begin{lstlisting}[style=diff, caption={Current broken HPX entry (c++.amazon.properties:5759--5767)}, label=lst:broken]
libs.hpx.name=HPX
libs.hpx.versions=v1110
libs.hpx.url=https://github.com/STEllAR-GROUP/hpx
libs.hpx.description=The C++ Standard Library for Concurrency and Parallelism
libs.hpx.packagedheaders=true
libs.hpx.versions.v1110.version=1.11.0
libs.hpx.versions.v1110.path=/app/hpx/hwloc_installed/include:/opt/compiler-explorer/libs/boost_1_84_0
libs.hpx.versions.v1110.libpath=/app/hpx/hwloc_installed/lib
libs.hpx.versions.v1110.liblink=hpx_wrap:hpx_init:hpx:hpx_core:hwloc:dl:rt
\end{lstlisting}

\subsubsection*{Bug \#1: Hardcoded Boost Path}

\textbf{What users see:} \code{fatal error: boost/version.hpp: No such file or directory}.

\textbf{Root cause:} Line~5765 references
\code{/opt/compiler-explorer/libs/boost\_1\_84\_0}.  When CE updated to
Boost~1.86.0, the path became stale.  The entry should use \code{\%DEP0\%}
variable interpolation. CE's \cefile{library\_builder.py} calls
\code{expand\_make\_arg()} to replace \code{\%DEP0\%} with the resolved
Boost path at build time.

% [GROUNDED: cmake/HPX_SetupBoost.cmake:39 — Boost_MINIMUM_VERSION = 1.71]
\textbf{Fix:} In \code{libraries.yaml}: add
\code{depends: [libraries/c++/boost\_bin 1.86.0]} and use
\code{-DBOOST\_ROOT=\%DEP0\%}.  HPX requires Boost~$\geq$1.71
(\cefile{cmake/HPX\_SetupBoost.cmake:39}); CE ships 1.86.0.

\subsubsection*{Bug \#2: Dynamic Linking in a Static-Only Sandbox}

\textbf{What users see:}
\code{error while loading shared libraries: libhpx\_core.so: cannot open}.

\textbf{Root cause:} The entry uses \code{liblink} (shared linking).
CE's \code{nsjail} sandbox (\cefile{etc/nsjail/user-execution.cfg}) mounts
\code{/opt/compiler-explorer} read-only and does not augment
\code{LD\_LIBRARY\_PATH} for HPX's \code{.so} files.  All successfully
integrated compiled libraries (Kokkos, Abseil, re2) use \code{staticliblink}.

\textbf{Fix:} \code{lib\_type: static} in YAML; \code{staticliblink} in
properties; \code{-DHPX\_WITH\_STATIC\_LINKING=ON} in CMake args.

\subsubsection*{Bug \#3: Only v1.11.0, Outdated Target}

\textbf{What users see:} Even if linking worked, v1.11.0 lacks the
\code{godbolt-minimal} CMake preset and \cefile{sandbox.hpp}.

% [GROUNDED: CMakeLists.txt:69-71 — HPX_VERSION = 2.0.0]
\textbf{Fix:} Target v2.0.0 (current release, per \cefile{CMakeLists.txt:69--71})
as primary; retain v1.11.0 as fallback.

\subsubsection*{Bug \#4: No \code{skip\_compilers}}

\textbf{What users see:} Nightly builds attempted on GCC~4.x / Clang~3.x
(no C++17, let alone C++20), wasting build capacity.

% [GROUNDED: .github/workflows/ — CI uses Clang 20, GCC 11+]
\textbf{Fix:} \code{skip\_compilers} listing all CE compiler IDs for
GCC~$\leq$11 and Clang~$\leq$14, e.g.\ \code{g101}, \code{g111},
\code{clang1400}.  CE's build system matches IDs exactly (not regex).
Derived from HPX's CI matrix (Clang~20, GCC~12+ for reliable C++20).
The full list will be refined in W2 against CE's live compiler inventory.

\subsubsection*{Bug \#5: No Default Example}

\textbf{What users see:} Empty editor.  No indication that
\code{hpx\_main.hpp} is required for runtime initialisation.

\textbf{Fix:} \code{example=hpx-sandbox} in properties, pointing to a
\code{transform\_reduce} example with \code{sandbox.hpp} benchmarking.

\subsubsection*{Bug \#6: \code{sandbox.hpp} Undiscoverable}

% [GROUNDED: libs/full/include/include/hpx/experimental/sandbox.hpp]
\textbf{What users see:} HPX ships a CE-specific header
(\cefile{libs/full/include/include/hpx/experimental/sandbox.hpp}%
\footnote{The doubled \code{include/include} is HPX's actual directory
structure: the outer \code{include/} is the CMake install-tree source
and the inner \code{include/} mirrors the public header prefix.}) with
environment introspection, benchmarking, and scaling telemetry, but
no one knows it exists.

\textbf{Fix:} The default example (Bug~\#5) will \code{\#include
<hpx/experimental/sandbox.hpp>}.

% ── Proposed Solution ─────────────────────────────────────────────────────────
\subsection*{Proposed Solution}

Following the project description, \textit{``we will maintain our own fork of
Compiler Explorer and experiment with the HPX integration locally before making
the integration public through a Pull Request''}, and under the guidance of
mentors \textbf{Hartmut Kaiser} and \textbf{Giannis Gonidelis}, I propose a
\textbf{three-repository, phased integration}:

\begin{description}[leftmargin=1.5em, itemsep=3pt]
  \item[PR\,\#1: \code{STEllAR-GROUP/hpx}]
    Refine the \code{godbolt-minimal} CMake preset (already in
    \cefile{CMakePresets.json}), improve \code{sandbox.hpp}, add integration
    documentation (\cefile{CONTRIBUTING\_CE.md}).

  \item[PR\,\#2: \code{compiler-explorer/infra}]
    Fix \code{libraries.yaml}: static linking, \code{\%DEP0\%} for Boost,
    v2.0.0 target, \code{skip\_compilers}, Ninja, \code{FETCH\_ASIO}.

  \item[PR\,\#3: \code{compiler-explorer/compiler-explorer}]
    Add \code{c++.amazon.properties} entries with correct metadata, link
    flags, and default example.
\end{description}

\textbf{Fork strategy.}
All development happens on a personal fork of \code{compiler-explorer/infra}
and \code{compiler-explorer/compiler-explorer}; the fork carries only the
HPX-specific YAML entry, properties additions, and default example; no
modifications to CE core code.
The fork is rebased onto CE \code{main} weekly (or before each draft PR
update) to prevent drift.
If upstream review extends beyond GSoC, the fork remains a
self-contained, deployable fallback that STE\texttt{||}AR Group can host
as an HPX-branded instance with a single \code{git clone}.

Each PR is self-contained.  Draft PRs open in Week~5 for early CE maintainer
feedback, providing 5+ weeks of review buffer before final evaluation.

\textbf{Upstream PR vs.\ HPX-branded instance.}
The project description mentions two approaches: integrating HPX into
the public CE instance and maintaining an HPX-branded fork.  The primary
deliverable is the upstream PR path (PR\,\#1--\#3 above), because it
reaches CE's full 3.5~million user base with zero maintenance overhead
for STE\texttt{||}AR Group.  The HPX-branded instance, with extended
timeouts, curated examples, and potential task-graph visualisation, is
a stretch goal (Section~``Stretch goals'').  If upstream PRs stall in
review, the fork doubles as a working fallback that STE\texttt{||}AR
Group can host independently.

\subsection*{Non-Obvious Issues Discovered During PoC}

Three non-obvious bugs surfaced that are invisible from reading
documentation alone:

\begin{enumerate}[leftmargin=2em, itemsep=2pt]
  \item \textbf{Boost headers on include path:} HPX headers transitively
        include \code{<boost/version.hpp>}.  CE must add Boost's include
        directory alongside HPX's.  The \code{\%DEP0\%} mechanism handles
        this in production; for the PoC, I added \code{/opt/homebrew/include}.

  \item \textbf{\code{hpx\_main.hpp} entry point wrapping:} HPX's
        zero-boilerplate mode requires the linker to wrap \code{main()}.
        Linux: \code{-Wl,-wrap=main}.  macOS:
        \code{-Wl,-e,\_initialize\_main}.  Without this flag:
        \code{Attempting to use hpx\_main.hpp functionality without
        linking to libhpx\_wrap}.

  % [GROUNDED: libs/CMakeLists.txt:80 (hpx_full → libhpx), :93 (hpx_full→hpx_core), :152 (hpx INTERFACE)]
  \item \textbf{Static library link order:} HPX's libraries have strict
        dependency order: \code{hpx\_wrap} $\to$ \code{hpx\_init} $\to$
        \code{hpx} $\to$ \code{hpx\_core}.  The \code{hpx} library
        (built from the \code{hpx\_full} target, \cefile{libs/CMakeLists.txt:80})
        contains the full runtime and links to \code{hpx\_core}
        (\cefile{libs/CMakeLists.txt:93}).  Reversing this order or omitting
        \code{hpx} produces undefined-reference errors.
        The \code{staticliblink} property must list them in this sequence.
\end{enumerate}

\textbf{Why \code{DISTRIBUTED\_RUNTIME=OFF} is a hard requirement, not an
optimisation:}  HPX's distributed runtime initialises AGAS (Active Global
Address Space) and parcelport networking.  AGAS binds a TCP socket for
the naming service; parcelports use TCP or MPI for inter-node communication.
The \code{nsjail} sandbox provides no network namespace and no socket
creation.  Without \code{DISTRIBUTED\_RUNTIME=OFF}, the HPX runtime would
attempt to bind a socket, receive \code{EACCES}, and abort with
\code{hpx::detail::throw\_exception()}.

\textbf{Memory layout of a statically linked HPX binary:}
\begin{itemize}[leftmargin=2em, itemsep=1pt]
  \item \code{.text} (HPX runtime + user code): $\sim$4--8\,MiB
  \item \code{.rodata} (string tables, vtables): $\sim$1--2\,MiB
  \item \code{.data/.bss} (global state): $\sim$0.5\,MiB
  \item Per-thread stack (default 2\,MiB each, 4 threads): $\sim$8\,MiB
  \item Per-thread heap (task queues, caches): $\sim$2--5\,MiB each
  \item \textbf{Total peak:} $\sim$35\,MiB of 200\,MiB budget
\end{itemize}

\subsubsection*{6. Execution Flow}

When a user selects HPX and clicks ``Run'':

\begin{enumerate}[leftmargin=2em, itemsep=1pt]
  \item Frontend sends code + compiler ID + library selection via POST
        to \code{/api/compiler/<id>/compile}.
  \item \cefile{options-handler.ts} resolves HPX's version and path from
        \code{c++.amazon.properties}.
  \item \cefile{base-compiler.ts} appends flags:
        \code{-I .../include -L .../lib -lhpx\_wrap -lhpx\_init
        -lhpx -lhpx\_core -pthread -DHPX\_APPLICATION\_EXPORTS -Wl,-wrap=main}
  \item Compiler produces a statically linked ELF ($\sim$8--15\,MiB).
  \item Binary executes inside \code{nsjail}; \code{hpx\_main.hpp}
        bootstraps the runtime.
  \item User code runs; output captured and displayed.
\end{enumerate}

\subsubsection*{7. Default Example}

\begin{lstlisting}[language=C++, caption={Default CE example for HPX}]
// HPX: The C++ Standard Library for Parallelism and Concurrency
// Try changing 'par' to 'seq' to compare performance!

#include <hpx/hpx_main.hpp>
#include <hpx/algorithm.hpp>
#include <hpx/numeric.hpp>
#include <hpx/experimental/sandbox.hpp>
#include <iostream>
#include <numeric>
#include <vector>

int main()
{
    hpx::experimental::sandbox::describe_environment(std::cout);

    constexpr std::size_t N = 1'000'000;
    std::vector<double> data(N);
    std::iota(data.begin(), data.end(), 1.0);

    auto report = hpx::experimental::sandbox::benchmark(
        "transform_reduce (1M doubles)",
        [&]() {
            volatile double r = hpx::transform_reduce(
                hpx::execution::seq, data.begin(), data.end(),
                0.0, std::plus<>{}, [](double x) { return x * x; });
            (void)r;
        },
        [&]() {
            volatile double r = hpx::transform_reduce(
                hpx::execution::par, data.begin(), data.end(),
                0.0, std::plus<>{}, [](double x) { return x * x; });
            (void)r;
        }
    );
    report.print(std::cout);
}
\end{lstlisting}

% [GROUNDED: libs/full/init_runtime/src/hpx_init.cpp:46-62 — DISTRIBUTED_RUNTIME guards]
\textbf{\code{HPX\_WITH\_DISTRIBUTED\_RUNTIME=OFF}:}
Prevents AGAS from binding a TCP socket at init and disables parcelport
registration (\cefile{hpx\_init.cpp:121--122}, guarded by
\code{HPX\_HAVE\_DISTRIBUTED\_RUNTIME} at line~46).  Mandatory in
\code{nsjail}'s network-isolated sandbox.  Also removes all action/component
registration code, reducing both binary size and compile time.

% [GROUNDED: CMakeLists.txt:468,2714-2717 — HPX_WITH_TOOLS]
\textbf{\code{HPX\_WITH\_TOOLS=OFF}:}
Disables the \cefile{tools/} subdirectory (\cefile{CMakeLists.txt:2714--2717}),
which contains development utilities (\code{hpxdep}, \code{inspect},
\code{hpx-top.py}, performance test scripts).  These are never used on CE.

\textbf{\code{HPX\_WITH\_CXX\_STANDARD=20}:}
Required for \code{hpx::experimental::sandbox} (uses C++20 concepts and
\code{std::chrono} duration formatting) and matches the \code{-std=c++20}
flag passed to user code.
